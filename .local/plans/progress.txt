## Stdin Prompt Reading for Ask Command (Story 10)

Added support for reading the prompt from stdin when no argument is provided.

### Changes Made

1. **New function `resolve_ask_prompt()`** (`crates/ralph/src/main.rs:711-749`):
   - Handles prompt resolution from multiple sources
   - When argument is provided: uses existing `classify_prompt_source()` for "-", file path, or inline
   - When argument is None: checks `stdin.is_terminal()` to detect piped input
   - If stdin is piped (non-terminal): reads from stdin
   - If stdin is interactive (terminal): returns error with usage hint
   - Validates prompt is non-empty after resolution

2. **execute_ask() updated** (`crates/ralph/src/main.rs:517`):
   - Now uses `resolve_ask_prompt()` instead of `args.prompt.unwrap_or_default()`

3. **Tests added** (`crates/ralph/src/main.rs:909-952`):
   - `test_resolve_ask_prompt_inline` - inline prompt works

   - `test_resolve_ask_prompt_empty_string_error` - empty string errors
   - `test_resolve_ask_prompt_whitespace_only_error` - whitespace-only errors
   - `test_resolve_ask_prompt_from_file` - file path reading works

### Usage

```bash
# Standard usage with argument
ralph ask 'what is 2+2'

# Read prompt from stdin (piped)
echo 'what is 2+2' | ralph ask
cat prompt.txt | ralph ask

# Explicit stdin with "-"
ralph ask -

# File path (existing behavior)
ralph ask /path/to/prompt.txt

# Error cases
```

### Key Design Decisions

- Uses existing `classify_prompt_source()` and `read_from_source()` for consistency
- Detects piped stdin via `std::io::stdin().is_terminal()` (false when piped)
- Error messages include usage hints for better UX
- Validates trimmed prompt for emptiness but preserves original whitespace in result
- Note: Unit testing stdin behavior is limited since `is_terminal()` returns false in test runners

Added `--permission-mode` flag with full configuration support (CLI, env var, config file).

### Changes Made

   - Added `permission_mode: String` field with `#[arg(long, value_name = "MODE", default_value = "bypassPermissions")]`
   - Doc comments describe all four modes and security implications
   - Default is `bypassPermissions` to allow tools to execute

2. **AskConfig updated** (`crates/ralph/src/ask.rs:68`):
   - Added `permission_mode: String` field
   - Added `DEFAULT_PERMISSION_MODE` constant set to `"bypassPermissions"`
   - Added `RALPH_PERMISSION_MODE_ENV` constant for env var name

3. **resolve_permission_mode() function** (`crates/ralph/src/ask.rs:69-92`):
   - Resolves permission mode with precedence: CLI > env var > config file > default
   - Checks `RALPH_PERMISSION_MODE` environment variable
   - Loads config from `~/.config/ralph/config.toml` `[ask]` section
   - Returns the CLI value if explicitly set (differs from default)

4. **AppConfig extended** (`crates/ralph/src/config.rs:81-99`):
   - Added `AskSection` struct with `permission_mode: Option<String>` field
   - Added `ask: AskSection` field to `AppConfig`
   - Both use serde default/skip_serializing_if for backward compatibility

5. **build_command() updated** (`crates/ralph/src/ask.rs:136`):
   - Now accepts `permission_mode` parameter
   - Inserts the mode into the claude command string

6. **execute_ask() updated** (`crates/ralph/src/main.rs`):
   - Both `execute_ask()` and `execute_ask_core()` call `resolve_permission_mode()`
   - Resolved value passed to `AskConfig.permission_mode`

### Permission Modes

- **default**: Requires approval for all tools
- **acceptEdits**: Auto-accepts file edits, requires approval for other tools
- **plan**: Read-only mode, no tools can modify files
- **bypassPermissions**: Auto-accepts all tool executions (default for ask command)

### Usage

```bash
# Default: bypassPermissions allows tool execution
ralph ask 'run ls'

# Override via CLI flag
ralph ask --permission-mode acceptEdits 'run ls'
ralph ask --permission-mode default 'run ls'
ralph ask --permission-mode plan 'read this file'

# Override via environment variable
RALPH_PERMISSION_MODE=acceptEdits ralph ask 'run ls'

# Override via config file (~/.config/ralph/config.toml)
# [ask]
# permission_mode = "acceptEdits"
```

### Resolution Precedence

1. CLI flag (`--permission-mode`) - highest priority
2. Environment variable (`RALPH_PERMISSION_MODE`)
3. Config file (`~/.config/ralph/config.toml` under `[ask]` section)
4. Default value (`bypassPermissions`) - lowest priority

### Key Design Decisions

- Default is `bypassPermissions` because `ask` is meant for single-shot tool execution
- CLI flag uses `Option<String>` to distinguish "not specified" from "explicitly set to default"
- If CLI is `Some(value)`, that value is used (highest priority) regardless of what the value is
- If CLI is `None`, env var and config file are checked in order
- Config file uses `[ask]` section to group ask-specific settings
- Security warning in `--help` text about `bypassPermissions` mode
- Backward compatible: old config files without `[ask]` section work fine

### Key Design Decisions

- `--clone-session` requires `--session` or `--continue` to specify source
- Clone always creates a NEW session with auto-generated slug
- Original session remains unchanged
- New session's metadata includes `cloned_from` field for traceability
- Conversation history is loaded from source and prepended to prompt
- Only the user's new prompt is stored in the iteration log (not history)
- Clone mode and continuation mode are mutually exclusive at runtime
