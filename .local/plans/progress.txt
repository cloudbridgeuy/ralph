## Progress Summarization Permission Fix (2026-01-19)

**Story**: Ensure prompt.txt summarizer invokes Claude with read/write permissions so summarization succeeds

**Problem**:
The default summarization command was `claude -p {prompt}` which ran Claude without any permission mode. Without explicit permission bypass, Claude CLI may prompt interactively for permission confirmations, which blocks the non-interactive subprocess.

**Changes**:
- Updated `SUMMARIZE_COMMAND` default from `"claude -p {prompt}"` to `"claude --dangerously-skip-permissions -p {prompt}"`
- Added test `test_default_summarize_command_has_permission_bypass` to verify the permission flag
- Updated `test_build_summarize_command_default_templates` to reflect new command format

**Key files**:
- `crates/core/src/context.rs` - Updated `SUMMARIZE_COMMAND` constant
- `crates/ralph/src/summarize.rs` - Updated test assertion

**Rationale**:
The summarization workflow doesn't require Claude to access files directly - it receives the progress file content as part of the prompt and outputs the summary to stdout. The Rust code then writes the output to the file. However, Claude CLI without explicit permission mode may still prompt interactively, which fails in a subprocess. Using `--dangerously-skip-permissions` ensures the subprocess completes without blocking.

## Persist Output Blocks on SIGTERM/SIGINT Interruption (2026-01-19)

**Story**: Persist output blocks on SIGTERM or SIGINT interruption

**Problem**:
When an interrupt signal (Ctrl+C / SIGINT or SIGTERM) occurred during subprocess execution, the partial output blocks accumulated by the StreamProcessor were being discarded. The `SubprocessError::Interrupted` variant contained a `partial_result` with the accumulated data, but this was being ignored when converting to `RunError::Interrupted`.

**Changes**:
- Extended `RunError::Interrupted` variant with two new fields:
  - `partial_result: Option<Box<StreamingSubprocessResult>>` - Contains output blocks accumulated before interrupt
  - `pending_before: Option<usize>` - Number of pending stories at iteration start (for partial log)
- Updated `invoke_with_failure_recovery` to capture and propagate the partial result from subprocess interrupts
- Updated the main run loop to handle interrupts during subprocess execution by propagating session context
- Added logic in `main.rs` `execute_run_with_prompting` to write a partial iteration log when an interrupt occurs during subprocess execution
- Added `into_option()` method to `LogMetadata` for clean conditional serialization

**Key files**:
- `crates/ralph/src/run/mod.rs` - Extended `RunError::Interrupted`, updated interrupt handling
- `crates/ralph/src/main.rs` - Added partial iteration log writing on interrupt
- `crates/ralph/src/iteration/metadata.rs` - Added `into_option()` method

**Data flow on interrupt**:
1. Signal handler sets `INTERRUPTED` flag
2. `invoke_subprocess_with_spinner` detects flag, drains output, calls `processor.finish()`
3. Returns `SubprocessError::Interrupted { partial_result }` with accumulated output blocks
4. `invoke_with_failure_recovery` converts to `RunError::Interrupted` preserving partial result
5. Main run loop fills in session context (slug, iterations, pending_before)
6. `execute_run_with_prompting` writes partial iteration log to session TOML
7. Session is finalized as "interrupted"

**Behavior**:
- When `ralph run` is interrupted during subprocess execution, a partial `iteration-N.toml` is written
- The partial log contains all output blocks accumulated up to the point of interruption
- Partial sessions can be replayed with `ralph replay <session-slug>`, showing output up to interruption
- Interrupts at other points (start of iteration, after log write) continue to work as before

**Acceptance criteria met**:
- Signal handlers are installed for SIGTERM and SIGINT (via ctrlc crate, already present)
- On signal, the accumulated Vec<OutputBlock> is serialized before exit
- Partial sessions can be replayed showing blocks up to the interruption

## Capture Specialized ToolResultVariant During Live Execution (2026-01-20)

**Story**: Capture specialized ToolResultVariant during live execution instead of Default

**Problem**:
Previously, Read, Grep, Glob, and TodoWrite tool results were all being serialized with the `Default` variant, discarding valuable metadata like line counts, match counts, and file counts. This metadata was available during live execution (used by verbose formatters) but not captured for replay.

**Changes**:
- Added specialized handlers for Read, Grep, Glob, and TodoWrite in `event_handler.rs` match statement (lines 284-341)
- Imported previously unused block builders: `build_read_result_block`, `build_grep_result_block`, `build_glob_result_block`, `build_todowrite_result_block`
- Removed dead code: `build_write_new_file_block` and `build_write_overwrite_block` (redundant with `build_write_result_block`)
- Each handler extracts metadata from tool results:
  - Read: file_path, line_count, truncated flag
  - Grep: match_count, output_mode, content
  - Glob: file_count, content, truncated flag
  - TodoWrite: message from result

**Key file**: `crates/ralph/src/stream_processor/event_handler.rs`

**Metadata extraction logic**:
- `line_count`: `content.lines().count()`
- `match_count`: `content.lines().filter(|l| !l.is_empty()).count()`
- `file_count`: Same as match_count for Glob results
- `truncated`: Checks for "... (truncated)" or "Output truncated" in content
- `output_mode`: Extracted from invocation input, defaults to "files_with_matches"

**TOML format** (for Read tool result):
```toml
[[output_blocks]]
type = "tool_result"
tool_name = "Read"
is_error = false

[output_blocks.result_type]
type = "read"
file_path = "src/main.rs"
content = "fn main() { ... }"
line_count = 50
truncated = false
```

- Edit tool results serialize as EditBeforeAfter or EditDiff variants (already working)
- Write tool results serialize as WriteNewFile or WriteOverwrite variants (already working)
- Read tool results serialize as Read variant with content and line count ✓
- Grep/Glob results serialize with match counts and file lists ✓
- TodoWrite results serialize with message ✓
- Default variant only used for truly unknown tools ✓
- All 525 tests pass, cargo xtask lint passes
